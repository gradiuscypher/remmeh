# Phase 1 — UI Design Contract

> Visual and interaction contract for Phase 1: Foundation & Core Chat.
> Generated by gsd-ui-researcher. This is a Textual TUI application — all contracts
> apply to terminal/character-based layout, ANSI color, Textual CSS (`.tcss`), and
> widget composition. Web design system concepts (px sizing, shadcn, hex colors)
> are translated to their TUI equivalents throughout.

## Spacing Scale

Textual uses character cells, not pixels. The spacing scale below is expressed in
**cells** (columns/rows) and maps to Textual CSS properties (`padding`, `margin`,
`height`, `width`). All values are multiples of 1 cell (the TUI equivalent of the
4px web grid).

| Token | Cells | Textual CSS Example | Usage |
|-------|-------|---------------------|-------|
| xs | 1 | `padding: 0 1` | Icon gaps, inline label padding |
| sm | 2 | `padding: 1 2` | Compact element spacing, status bar padding |
| md | 4 | `padding: 2 4` | Default widget interior padding |
| lg | 6 | `margin: 3 0` | Section separation between major panels |
| xl | 8 | `padding: 4 8` | Layout-level gutters |
| 2xl | 12 | — | Major structural breaks (reserved) |
| 3xl | 16 | — | Page-level spacing (reserved) |

**Declared fixed heights (from REQUIREMENTS.md UI-01):**

| Region | Height | Source |
|--------|--------|--------|
| Input panel | 5 rows (3 content + 2 border) | REQUIREMENTS.md UI-01, SUMMARY.md |
| Status bar | 1 row | SUMMARY.md architecture |
| Chat view | `1fr` (fills remaining terminal height) | SUMMARY.md — "CSS `1fr` for ChatView height" |

**Exceptions:**
- Input panel is fixed at **5 rows** (3 visible lines + top/bottom border) — not derived from spacing scale
- Chat view uses `1fr` — fills all space not consumed by input panel and status bar
- Minimum terminal size assumed: 80 columns × 24 rows

## Typography

Textual renders in a monospace terminal — all text is fixed-width character cells.
"Size" in a TUI maps to **role and markup style** rather than pixel font-size.
Textual's `Markup` and `Markdown` widget handle visual hierarchy.

| Role | Textual Equivalent | Weight/Style | Line Height |
|------|-------------------|--------------|-------------|
| Body | Plain text in `RichLog` / `Markdown` | Normal (terminal default) | 1 (single-spaced — terminal default) |
| Label | `Static` widget with `[bold]` markup | Bold | 1 |
| Code | `Markdown` fenced code block (syntax-highlighted) | Monospace bold (Rich's `Code` style) | 1 |
| Heading | `Markdown` H1–H3 rendered by Textual `Markdown` widget | Bold + `#` prefix stripped | 1 |

**Rules:**
- Body text in `RichLog` during streaming: unstyled terminal default — no markup applied to streaming tokens
- Body text in `Markdown` on completion: standard Markdown rendering via Textual's built-in `Markdown` widget
- Code blocks: syntax-highlighted by Rich/Textual using language hint from fenced-code fence — always rendered in a distinct visual block
- Assistant messages use `Markdown` widget (completed) and `RichLog` (streaming) — these are the only two text rendering paths
- User messages use `Static` widget with `[bold]` markup applied to the message text — visually distinct from assistant

## Color

Textual apps use **named ANSI colors** and **CSS color names** in `.tcss` files.
The 60/30/10 rule is applied to terminal surface distribution (not hex values).

| Role | Color Token | Textual CSS Value | Usage |
|------|-------------|-------------------|-------|
| Dominant (60%) | `$background` | `$background` (Textual default dark) | Full-screen app background, chat view surface |
| Secondary (30%) | `$surface` | `$surface` (Textual default — slightly lighter) | Input panel background, status bar background |
| Accent (10%) | `$accent` | `$accent` (Textual default — blue) | See reserved-for list below |
| Streaming indicator | `$warning` | `$warning` (Textual default — yellow) | Streaming-in-progress indicator only |
| Error | `$error` | `$error` (Textual default — red) | Inline error messages in chat thread only |
| User message border | `$primary` | `$primary` (Textual default — slightly lighter than surface) | Left border or prefix on user messages |

> **Textual default theme:** Uses Textual's built-in dark theme (`DEFAULT_CSS` dark variant).
> No custom theme file for Phase 1. Custom theme is Phase 3+ scope.

**Accent (`$accent`) reserved for:**
1. The currently active/focused widget border (Textual default focus ring behavior)
2. The cursor position indicator in `TextArea` (input panel)
3. The model name label in the status bar (to signal it is tappable/actionable in future phases)

**Accent is NOT used for:**
- Message bubbles or chat content
- Static labels or headings
- Borders on non-focused widgets

**Error (`$error`) reserved for:**
- Inline error `MessageWidget`s in the chat thread (API errors, rate limits, timeouts)
- These appear as a distinct error-styled Static widget inserted into the chat view

## Layout Contract

> Source: REQUIREMENTS.md UI-01, SUMMARY.md architecture section

Phase 1 delivers a single full-screen layout with no panels toggling. The session
sidebar is Phase 2 scope — it must NOT be scaffolded in Phase 1 even as a placeholder.

```
┌─────────────────────────────────────────────────────────┐
│  Status Bar (1 row): model name left | app name right   │  ← $surface bg
├─────────────────────────────────────────────────────────┤
│                                                         │
│  ChatView (1fr, scrollable)                             │  ← $background bg
│  └─ MessageWidget × N                                   │
│     ├─ User messages: [bold]You:[/] + Static            │
│     └─ Assistant: RichLog (streaming) → Markdown (done) │
│                                                         │
├─────────────────────────────────────────────────────────┤
│  InputPanel (5 rows): TextArea, Enter=submit            │  ← $surface bg
│  Shift+Enter=newline, scroll when content > 3 lines     │
└─────────────────────────────────────────────────────────┘
```

**Widget inventory for Phase 1:**

| Widget Class | Textual Base | Purpose |
|---|---|---|
| `RemmehApp` | `textual.app.App` | Root app, keybindings, screen registry |
| `ChatScreen` | `textual.screen.Screen` | Primary layout, composes ChatView + InputPanel |
| `ChatView` | `VerticalScroll` | Scrollable chat thread container |
| `MessageWidget` | `Widget` | Single message: user or assistant |
| `InputPanel` | `Widget` wrapping `TextArea` | Bottom 5-row input; posts `Submitted` message |
| `StatusBar` | `Static` | 1-row status bar — model name + app name |

**What is NOT in Phase 1:**
- `SessionSidebar` — Phase 2
- `CommandPalette` providers — Phase 3 (Textual's built-in `CommandPalette` can be wired in Phase 3)
- Any overlay or modal screen

## Interaction Contract

> Source: REQUIREMENTS.md, SUMMARY.md pitfalls and architecture

### Keybindings (Phase 1 only)

| Key | Action | Scope | Notes |
|-----|--------|-------|-------|
| `Enter` | Submit message | InputPanel focused | `priority=False` — scoped to InputPanel only |
| `Shift+Enter` | Insert newline | InputPanel focused | Standard `TextArea` behavior |
| `Ctrl+C` / `Ctrl+Q` | Quit app | App-level | `priority=True` — fires even when InputPanel focused |
| `Escape` | Cancel streaming (if active) | App-level | Cancels `@work(exclusive=True)` stream worker |
| `PageUp` / `PageDown` | Scroll chat view | ChatView focused | Standard `VerticalScroll` behavior |

**Rule:** App-level bindings use `priority=True` ONLY for quit (`Ctrl+C`, `Ctrl+Q`) and
stream cancel (`Escape`). All other bindings are scoped to their widget to prevent
conflicts with `TextArea` text editing.

### Input Panel Behavior

- `TextArea` configured with 3 visible lines, vertical scroll enabled
- `Enter` key: consumed by `InputPanel`, posts `InputPanel.Submitted` custom message, clears `TextArea`
- `Shift+Enter`: inserts `\n` into `TextArea` — default `TextArea` behavior, no override needed
- Input panel border highlights with `$accent` when focused (Textual default focus ring)
- Input is disabled (read-only) while streaming is active — re-enabled on stream completion or error

### Streaming States

| State | Visual Indicator |
|-------|-----------------|
| Idle | No indicator — input panel enabled |
| Streaming | Status bar shows `⠋ Streaming…` spinner (Textual `LoadingIndicator` or Unicode spinner in `Static`) |
| Stream complete | Spinner removed; `RichLog` content swapped to `Markdown` widget in `MessageWidget` |
| Error | Inline error `MessageWidget` appended to `ChatView`; input panel re-enabled; no spinner |

**Streaming swap pattern (from SUMMARY.md Pitfall 2):**
1. While streaming: `RichLog.write(chunk)` per token — no scroll reset
2. On completion: replace `RichLog` with `Markdown(full_response_text)` in `MessageWidget`
3. Auto-scroll: `ChatView.scroll_end(animate=False)` called after each chunk write

### Auto-Scroll Contract

- Chat view auto-scrolls to bottom on every new chunk during streaming
- Auto-scroll is disabled if the user has manually scrolled up (detect with `VerticalScroll.is_at_end` or scroll position)
- On stream completion, auto-scroll to end is always restored

| Element | Copy |

**Error display rules:**
- All errors appear as inline `MessageWidget`s in the chat thread (REQUIREMENTS.md CHAT-05)

- Errors never use modal dialogs or overlays in Phase 1
- Destructive actions: **none in Phase 1** — no delete, no session clear, no data destruction

## Implementation Notes for Executor

> These notes resolve ambiguities that would otherwise cause implementation detours.

1. **Do NOT use `MarkdownViewer` widget** — it includes a TOC sidebar that is wrong for chat. Use `Markdown` widget only. (Source: SUMMARY.md explicit warning.)

2. **Do NOT use `height: auto` on `ChatView`** — it collapses to zero height. Use `height: 1fr`. (Source: SUMMARY.md Pitfall 8.)

3. **All LLM API calls MUST use `@work(exclusive=True)`** — never `await` directly in a message handler. Submitting a new message cancels any in-progress stream. (Source: SUMMARY.md Pitfall 1.)

4. **Set `exit_on_error=False` on every `@work` decorator** — default `exit_on_error=True` kills the TUI on any API error. Handle `WorkerState.ERROR` to show inline error widget. (Source: SUMMARY.md Pitfall 3.)

5. **SQLite WAL mode: enable at startup** — `PRAGMA journal_mode=WAL` before any reads/writes. (Source: SUMMARY.md Pitfall 4.)

6. **`python-dotenv` must load before `App.__init__` runs** — call `load_dotenv()` at module import time in `main.py`. (Source: SUMMARY.md stack section.)

7. **`asyncio_mode = "auto"` in `pyproject.toml`** — set before writing the first test or async tests silently never run. (Source: SUMMARY.md Pitfall 5.)

8. **Streaming swap sequence** — use `RichLog.write(chunk)` per token during streaming; replace with `Markdown(full_text)` on completion. Do NOT call `Markdown.update()` per chunk. (Source: SUMMARY.md Pitfall 2.)

9. **`InputPanel` consumes `Enter` key** — use `TextArea`'s key handling or override `on_key` in `InputPanel`; ensure the app-level binding does NOT intercept Enter when focus is in the input panel.

10. **Phase 1 uses a hardcoded default model** — dynamic model list is Phase 2. Hard-code `DEFAULT_MODEL = "anthropic/claude-3.5-sonnet"` as a module constant; display it in status bar. Phase 2 will replace this with dynamic fetch.
